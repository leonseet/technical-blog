Why your agent eval harness is lying to you

Jul 12, 2026 8 min read agentsevals

Every agent project I've worked on starts the same way: someone builds an eval set, reports a pass rate, and everyone relaxes. The number goes up, the demos work, and then production quietly falls apart in ways the harness never saw coming.

The problem isn't that the evals are wrong. It's that a single aggregate number throws away the one thing you actually need — where and how the agent fails.

Pass rates compress too much

A pass rate of 82% feels informative. It isn't. Two agents at 82% can fail in completely different ways: one flakes on long-horizon tasks, the other confidently hallucinates tool arguments. You ship them as equals and find out the difference the hard way.

An aggregate score is a summary of a distribution you never looked at.

Instrument the trajectory, not the outcome

Instead of scoring the final answer, score each step. Log the tool call, the arguments, and whether the step moved the task forward. A small wrapper is enough:

// score each step, not just the final answer
function scoreTrajectory(steps) {
  return steps.map((step, i) => ({
    index: i,
    tool: step.tool,
    valid: isValidArgs(step.tool, step.args),
    advanced: step.stateDelta > 0,
  }));
}

Now a failure isn't "task failed" — it's "step 4 called search with an empty query." That's a bug you can actually fix.

What to log

Keep it boring: inputs, outputs, a timestamp, and a stable trajectory id. You want to be able to grep for a failure class months later.

The takeaway

Aggregate metrics are for dashboards. When you're actually improving an agent, look at the trajectories. The harness should make failure legible, not tidy.

Spec at a glance

The concrete decisions this mockup encodes. Everything here is up for discussion — that's what this ticket is for.

Typography

Geist SansUI · headings
Literatapost body (reading serif)
Geist Monocode · dates · routes
  • Sans headings + serif body — contrast-axis pairing.
  • Literata: screen-optimized reading serif (not the over-used Newsreader/Lora/Fraunces). Alt: Source Serif 4.
  • All self-hosted via next/font (zero layout shift).
  • Reading measure ~68ch; serif body ~18px/1.72.

Color

  • Strategy: restrained — neutral surfaces, one accent.
  • shadcn neutral base ramp, tokens in OKLCH.
  • Light: pure white; Dark: true near-black.
  • Accent used only for links, active nav, focus, hover.
  • Accent: Indigo — oklch(0.52 0.18 265) light, oklch(0.72 0.15 265) dark.

Dark / light

  • next-themes, class strategy.
  • Default system; header toggle overrides.
  • No flash: inline script sets theme pre-paint.
  • Both themes tuned, not naively inverted.

shadcn/ui

  • Style new-york, base color neutral.
  • Components: button, dropdown-menu (mobile nav + theme), badge (tags), separator, avatar, aspect-ratio.
  • Post rows are plain links, not cards.
  • No heavy blocks — hand-composed layout.

Aesthetic

  • Reading-first, generous whitespace, real hierarchy.
  • Deliberately not editorial-serif/italic (over-used).
  • Quiet chrome; the writing is the loud part.
  • Radii ≤ 14px; hairline borders; soft shadows.

Prose / MDX

  • Code: Shiki via rehype-pretty-code (matches theme).
  • Headings get anchor links (rehype-autolink).
  • Diagrams = images via next/image.
  • Sticky TOC on desktop only; inline code chips.
